# Provenance Guard

A backend content attribution system for creative platforms. Provenance Guard
classifies submitted text as likely AI-generated or likely human-written,
returns a confidence score, displays a transparency label, and provides an
appeals workflow for contested classifications.

---

## Architecture

A submitted piece of text travels through the following components:

1. **Validation** — The `/submit` endpoint checks that the request includes
   both `text` and `creator_id`, and that the text meets a minimum length
   of 50 characters.
2. **Detection Pipeline** — Two independent signals analyze the text in
   sequence: Groq LLM Classification (semantic analysis) and Stylometric
   Heuristics (structural analysis).
3. **Confidence Scoring** — The two signal scores are combined using a
   weighted average (Groq × 0.6 + Stylometric × 0.4) into a single
   confidence score between 0.0 and 1.0.
4. **Transparency Label** — The confidence score maps to one of three
   plain-language labels shown to the user.
5. **Audit Log** — Every decision is recorded as a structured JSON entry
   with content ID, timestamp, both signal scores, final confidence,
   attribution result, and appeal status.

For the appeal flow: a creator submits `POST /appeal` with their
`content_id` and reasoning. The system updates the entry's status to
`under_review` and logs the appeal reasoning alongside the original decision.

## Detection Signals

### Signal 1: Groq LLM Classification (weight: 0.6)

The text is sent to `llama-3.3-70b-versatile` with a prompt asking it to
return a single float between 0.0 and 1.0 representing AI likelihood.

**Why this signal:** LLM-based classification captures semantic and stylistic
patterns holistically — things like overly smooth transitions, complete
sentence structure, and "safe" word choices that are hard to quantify but
easy for a language model to recognize.

**Why higher weight:** In testing, the Groq signal showed stronger
differentiation between clearly AI and clearly human text. It is the more
reliable signal and receives 60% of the combined score.

**Blind spot:** The model may be biased toward its own training distribution.
Text generated by a different LLM (e.g., GPT-4) may not trigger the same
recognition patterns. AI content that has been heavily edited by a human
may also evade detection.

### Signal 2: Stylometric Heuristics (weight: 0.4)

Three structural metrics are computed in pure Python:
- **Sentence length variance** — AI writing tends to be uniform in sentence
  length; human writing varies naturally. Low variance scores higher for AI.
- **Type-token ratio** — Vocabulary diversity relative to total word count.
  Higher diversity is weighted toward AI likelihood.
- **Punctuation density** — AI tends to use punctuation moderately and
  consistently. Scored on a normalized scale.

**Why this signal:** Stylometric features are independent of the LLM signal —
they measure structure, not semantics. This means the two signals can
disagree meaningfully, which produces more honest uncertainty scores.

**Blind spot:** Professional writers who deliberately write with consistent
structure (uniform sentence length, formal punctuation) may score high on
AI likelihood despite being human authors.

### Confidence Scoring
```
Final score = (Groq score × 0.6) + (Stylometric score × 0.4)
```

Thresholds are intentionally asymmetric to reduce false positives —
mislabeling a human creator's work as AI is worse than missing an
AI-generated piece:

| Score Range | Attribution |
|---|---|
| > 0.70 | likely_ai |
| 0.40 – 0.70 | uncertain |
| < 0.40 | likely_human |

**Example — high confidence AI:**
```json
{
  "text": "Artificial intelligence represents a transformative paradigm...",
  "groq_score": 0.8,
  "stylometric_score": 0.551,
  "confidence": 0.7004,
  "attribution": "likely_ai"
}
```

**Example — high confidence human:**
```json
{
  "text": "ok so i finally tried that new ramen place downtown...",
  "groq_score": 0.2,
  "stylometric_score": 0.348,
  "confidence": 0.2592,
  "attribution": "likely_human"
}
```

## Transparency Labels

All three label variants are displayed to users in plain language.
The exact text for each variant:

**High-confidence AI (score > 0.70):**
> "Our system believes this content was created by AI. Confidence: High.
> If you are the creator and believe this is incorrect, you may submit
> an appeal."

**Uncertain (score 0.40 – 0.70):**
> "Our system was unable to confidently determine whether this content
> was written by a human or AI. If you are the creator, you may submit
> an appeal to provide more context."

**High-confidence Human (score < 0.40):**
> "Our system believes this content was created by a human. Confidence:
> High. If you believe this assessment is incorrect, you may submit
> an appeal."

## Appeals Workflow

Any creator who disagrees with their classification result can submit
an appeal via `POST /appeal` with their `content_id` and
`creator_reasoning`.

The system:
1. Locates the original audit log entry by `content_id`
2. Updates `status` from `"classified"` to `"under_review"`
3. Adds `appeal_reasoning` to the log entry

Automated re-classification is not performed. A human reviewer would
see the full log entry including original confidence scores, both signal
scores, attribution result, and the creator's reasoning.

**Test appeal:**
```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "f427ef5e-c444-447e-b159-27ed45f3223b",
    "creator_reasoning": "I wrote this myself from personal experience.
    I am a non-native English speaker and my writing style may appear
    more formal than typical."
  }' | python -m json.tool
```

**Result:**
```json
{
  "content_id": "f427ef5e-c444-447e-b159-27ed45f3223b",
  "message": "Appeal received. Your submission is now under review.",
  "status": "under_review"
}
```

## Rate Limiting

Applied to `POST /submit`:
- **10 requests per minute**
- **100 requests per day**

**Reasoning:** A legitimate creator submitting their own work would
rarely exceed 10 submissions in a minute — most writing sessions involve
one or two submissions. The per-minute limit blocks automated scripts
flooding the system. The 100/day limit allows heavy legitimate use while
making large-scale scraping cost-prohibitive.

## Known Limitations

**1. Humanized AI content evades detection.**
AI-generated text passed through humanization tools (e.g., Quillbot,
Undetectable.ai) tends to exhibit higher sentence length variance and
vocabulary diversity. Both signals are calibrated against raw AI output —
humanized content disrupts the structural patterns the stylometric signal
relies on, and the semantic patterns the Groq signal looks for. This is
the most likely attack vector in a real deployment.

**2. Formal human writing scores as uncertain or AI.**
Academic writing, legal text, and professionally edited prose naturally
exhibit low sentence variance and consistent punctuation — the same
features the stylometric signal associates with AI. A legal brief written
entirely by a human may score 0.65–0.70, landing in the uncertain band
or triggering a likely_ai label.

## Spec Reflection

**Where the spec helped:** The planning.md threshold design (0.40 / 0.70
asymmetric bands) directly shaped the implementation. Writing out the
reasoning — that false positives are worse than false negatives on a
creative platform — meant the threshold decision was already made before
writing any code. When test results showed the AI text scoring 0.70
exactly, there was no ambiguity about whether to adjust the threshold or
accept the result.

**Where implementation diverged:** The original spec set the AI threshold
at 0.75. After testing four inputs in Milestone 4, the AI text scored
0.70 — just below the threshold — and was labeled uncertain. The threshold
was adjusted to 0.70 to better reflect actual signal behavior. The spec
was updated to match.

## AI Usage

**Instance 1 — Stylometric signal function:**
Provided the Detection Signals section of planning.md and the architecture
diagram, and asked for a Python function computing sentence length variance,
type-token ratio, and punctuation density returning a float between 0.0
and 1.0. The generated function used a fixed normalization divisor of 50
for variance, which was kept after manual testing showed it produced
reasonable scores across the test inputs.

**Instance 2 — Flask app skeleton and /submit route:**
Provided the API Endpoints section and architecture diagram, and asked for
a Flask app skeleton with a POST /submit stub. The generated output used
`request.json` instead of `request.get_json()`, which was corrected before
use. The route structure and validation logic matched the spec and were
kept as generated.

## Setup

```bash
git clone <your-repo-url>
cd provenance-guard
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Create a `.env` file:
```
GROQ_API_KEY=your_key_here
```
Run the server:
```bash
python app.py
```
